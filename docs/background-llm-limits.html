Background reading — optional People coming to AI tools for the first time often have one of two misconceptions: either they think AI is magic and can do anything, or they get frustrated because it can't do a specific thing they asked for. This page gives you the mental model that explains both.

What an AI Language Model Actually Is

Claude, ChatGPT, Copilot, and similar tools are all large language models (LLMs). At their core, they are extremely sophisticated pattern-matching engines trained on enormous amounts of text — books, websites, code, conversations.

When you ask a question, the model predicts, word by word, what the most useful response would look like. It's not "thinking" in the way a human does, and it doesn't "know" things the way a textbook does. It generates text that fits the pattern of a good response to your input, based on what it learned during training.

This is a subtle but important distinction. It explains both why these tools are so capable (they've learned from an enormous range of human knowledge) and why they sometimes produce confidently wrong answers.

The Key Insight: Claude Can't — But Code Can

There are many things that Claude (or any LLM by itself) genuinely cannot do: read a file on your computer, send an email, fetch live data from a website, make a reservation, or run a calculation on 10,000 rows of data. These require access to external systems, and Claude is just generating text.

But here's the insight that changes everything:

✗ Claude cannot do this

"Read my CSV file and calculate the average score by department."

→

✓ Claude can write code that does this

A Python script that reads your CSV, groups by department, and calculates the average — which you then run yourself.

Claude is a code generator. It can write programs that do things Claude itself cannot. Your job shifts from "write the code" to "describe what the code should do and run it." That's a very different — and often much more accessible — way to work.

Examples of the Pattern

What you want Can Claude do it directly? Can Claude write code that does it?
Sort a list of 10,000 names alphabetically No — too much data to process in text Yes — a 3-line Python script
Send an email on your behalf No — no access to your email Yes — a script using Gmail API or a mail library
Read a file from your hard drive No — unless you paste the contents in Yes — one line of Python: open("file.txt").read()
Check a website for a price change No — can't browse live web Yes — a script using the requests library
Explain what a piece of code does Yes — this is exactly what LLMs are great at —
Write a function from a description Yes — one of the most powerful use cases —
Debug an error message Yes — paste the error and the code, ask what's wrong —
The practical takeaway: when you hit something Claude "can't do," ask yourself — can it write code that does it? The answer is often yes. Then your job is to run that code and report back what happened.

Hallucinations: When AI Makes Things Up

What is a hallucination?

A hallucination is when an AI produces an answer that sounds confident and plausible but is factually wrong. It might cite a research paper that doesn't exist, invent a function name that looks real but isn't, or give you code that runs but produces wrong results.

It happens because the model is generating text that fits the pattern of a correct answer, not because it's verifying facts against a source of truth. It doesn't know it's wrong — it has no mechanism for self-doubt.

How common are hallucinations?

For common, well-documented tasks — writing a Python function, fixing a CSS layout, explaining a terminal command — hallucinations are rare because the model has seen correct examples of these thousands of times. For obscure questions, recent events, or very specific factual claims, the risk is higher.

Code is actually easier to verify than facts — you run it and see if it works. A hallucinated historical date is harder to catch.

How to handle hallucinations

  • Run the code. Don't assume it works because it looks right. Test it.
  • Verify specific claims. If Claude tells you a specific function exists in a library, check the documentation.
  • Ask follow-up questions. "Are you sure about that?" sometimes prompts a correction.
  • Use it as a starting point, not an endpoint. AI-generated code is a draft, not a final answer.
Never submit AI-generated work as your own without understanding it first. Not because it will always be wrong — often it's correct — but because you need to be able to stand behind it, explain it, and fix it when something changes.

The Right Mental Model: A Knowledgeable Collaborator

The most useful way to think about Claude and similar tools is as a knowledgeable collaborator who types very fast and never gets tired, but who:

  • Can only see what's in front of them (what you've shown in the conversation)
  • Sometimes fills in gaps with plausible-sounding guesses
  • Is excellent at well-documented, common tasks
  • Needs you to verify, test, and direct the work
  • Gets dramatically better when given clear, specific context

You are still the engineer. Claude handles the typing. That division of labor is what makes this workflow so productive — as long as you stay in the loop.
