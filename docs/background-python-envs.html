Background reading — optional This page explains something that confuses almost everyone at some point: why does python --version sometimes give a different answer than you expect, and why do projects ask you to set up a "virtual environment"? You don't need this to follow the setup guides, but it'll help if things ever seem inconsistent.

Python Is Not One Single Thing

When you install Python, you're installing a specific version — like Python 3.11 or 3.12. That installation includes the Python interpreter (the program that runs your code) plus a set of built-in modules. But here's the catch: you can install multiple versions of Python on the same machine, and they coexist independently.

On top of that, Python has a massive ecosystem of third-party packages — libraries like pandas, requests, flask — which you install separately using pip. These packages are installed into a specific Python installation, not globally on your computer.

This means two different Python installations on the same machine can have completely different sets of packages. Project A might need an older version of a library; Project B might need a newer one. If they share the same installation, one of them breaks.

What an Environment Is

A Python environment is an isolated container — its own Python interpreter plus its own set of packages. Code running inside that environment sees only its packages, not anything installed in other environments.

Think of it like different toolboxes for different jobs. The plumbing toolbox and the electrical toolbox are both in the garage, but you pick up the right one before starting work, and they don't interfere with each other.

If you install all your packages into the system Python and projects start conflicting, unraveling it is painful. Using environments from the start costs almost nothing and saves a lot of headaches.

Two Common Tools: venv and conda

venv (built into Python)

  • Ships with Python — no install needed
  • Lightweight: creates a folder in your project
  • Manages Python packages only
  • Activated per terminal session
  • Best for most Python projects

conda (Anaconda / Miniconda)

  • Separate install — heavier
  • Manages Python and non-Python dependencies
  • Popular in data science and scientific computing
  • Can install different Python versions per environment
  • Best when you need numpy, scipy, or non-Python libs
For this course, you don't need to set up either. The Python setup guide installs Python cleanly and you'll run scripts directly. This page is context for when you encounter venv or conda mentioned in a tutorial or project README.

The "Which Python" Problem

If you have multiple Python installations, the terminal picks one based on the PATH (the list of places the OS searches when you type a command — see Paths Explained for how that works). Which one gets picked depends on the order things were installed and whether any activation scripts have run.

This is why sometimes python returns version 3.9 but you thought you installed 3.12. Or why a package you're sure you installed doesn't seem to be there.

# Check which Python is actually being used
C:\Users\you> python --version
Python 3.9.7

# Find out where that Python actually lives
C:\Users\you> where python
C:\Users\you\AppData\Local\Programs\Python\Python39\python.exe
C:\Users\you\AppData\Local\Microsoft\WindowsApps\python.exe
The Microsoft Store version of Python (WindowsApps\python.exe) is a stub that redirects to the Store. It can cause confusing behavior. The Python setup guide shows you how to avoid it.

How VS Code Picks a Python Interpreter

When you open a Python file in VS Code, it shows the currently selected Python interpreter in the bottom-left corner of the window. Click on it to switch — VS Code will list all the Python installations and environments it can find on your machine.

If you're in a project with a venv folder, VS Code usually detects it automatically and offers to activate it. This means the terminal inside VS Code will use that environment's Python and packages, not the system one.

If a script runs fine in the terminal but VS Code shows errors, the interpreter is probably different from what the terminal is using. Check the bottom-left corner of VS Code and make sure it matches.

What You Need to Know for This Course

For these exercises, a single clean Python install is enough. You don't need environments yet. But now you have a mental model for what's happening when you see terms like venv, activate, conda env create, or (myenv) appearing in a terminal prompt.
