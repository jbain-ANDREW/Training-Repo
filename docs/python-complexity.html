Start Here — Do You Even Need This?

If you are just getting started with Python, you probably don't need any of this yet. A plain Python install is fine for writing and running scripts.

This page is for when you hit one of these situations:

Situation 1 — "It works on my machine but not yours" You share a script with a classmate. It runs fine for you but crashes for them because they have a different version of a package installed.
Situation 2 — Two projects need different package versions Project A needs numpy 1.24. Project B needs numpy 2.0. Installing one breaks the other.
Situation 3 — You installed a lot of packages and things are a mess Running pip list shows 80 packages and you're not sure which ones you actually need.

When you feel one of those pains, environments are the answer.

What Is a Package?

Python by itself can do a lot, but most real work relies on packages — libraries written by others that you install and import. Examples:

  • numpy — fast numerical arrays and math
  • matplotlib — plotting and graphs
  • pandas — data tables (like Excel, but in code)
  • scipy — scientific computing

You install packages with pip:

pip install numpy

By default, pip installs packages globally — into the single Python installation on your machine. That's fine when you have one project. It becomes a problem when you have many.

What Is a Virtual Environment (venv)?

A virtual environment is an isolated copy of Python with its own separate set of packages. When you activate one, pip install puts packages there instead of globally.

How it works in practice

# Create a virtual environment in a folder called .venv
python -m venv .venv

# Activate it (Windows)
.venv\Scripts\activate

# Your terminal prompt changes to show (.venv)
# Now pip installs go into .venv, not globally
pip install numpy

# When you're done
deactivate
VS Code detects .venv folders automatically and offers to use them. Once activated, every time you open a terminal in VS Code it will activate the environment for you.

Why it helps

Without venv

  • All packages shared globally
  • Two projects can't use different versions of the same package
  • Hard to know what a project actually needs
  • Upgrading one project can break another

With venv

  • Each project has its own packages
  • No version conflicts between projects
  • pip freeze lists exactly what this project needs
  • Safe to update or delete without affecting other projects

requirements.txt — sharing your environment

Once you have a working venv, you can export its package list:

pip freeze > requirements.txt

A classmate can then recreate your exact setup:

python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt

This is the standard way to share Python projects — commit requirements.txt to your repo, never commit the .venv folder itself.

What Is Conda?

Conda (via Anaconda or Miniconda) is an alternative to pip + venv. It manages both packages and Python versions, and it handles non-Python dependencies (like C libraries) that pip sometimes struggles with.

When Conda is the right choice

  • You work heavily with data science libraries (numpy, scipy, tensorflow)
  • Your instructor or course materials specify Conda
  • You need to switch between Python versions for different projects
  • You're on a machine where installing compiled packages with pip fails

When plain Python + venv is the right choice

  • You're new to Python and just getting started
  • Your project is general-purpose scripting or web work
  • You want a simpler mental model
  • You're sharing code that others will run without Conda
Mixing Conda and pip in the same environment can cause subtle problems. If you use Conda, prefer conda install for packages whenever possible, and only fall back to pip for packages not available in Conda channels.

What About pyenv?

pyenv is a tool for managing multiple Python versions on the same machine (e.g., Python 3.10 for one project, 3.12 for another). It is very common on Mac and Linux.

On Windows, there is a port called pyenv-win, but it requires extra setup and is generally not worth it for beginners. If you need multiple Python versions on Windows, Conda handles it more smoothly.

For most coursework, one well-configured Python install is enough. You only need pyenv if you are actively maintaining projects that require different Python versions simultaneously.

Summary — Which Tool Do You Need?

Just starting out Plain Python install from python.org. No venv needed yet.
Starting a project that installs packages Create a venv for that project. Commit a requirements.txt.
Heavy data science / ML work Consider Miniconda. Follow your course instructions.
Need multiple Python versions Use Conda environments, or ask your instructor.
