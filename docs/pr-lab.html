Pull Requests — Why They Exist

The core problem: everyone editing the same codebase

On a real project, multiple people are writing code at the same time. If everyone pushed directly to main, you would constantly overwrite each other's work, break things that were just fixed, and have no way to review changes before they went live.

The solution is branches. Each contributor works on their own branch — a private copy of the project where changes are isolated. When the work is ready, they open a Pull Request (PR): a formal request to merge their branch into main.

mainlive code
→ branch →
your branchyour changes
→ PR →
mainmerged

Branch → make changes → PR → review → merge back into main

What a Pull Request actually is

A PR is not a Git concept — it is a GitHub feature layered on top of Git. When you open a PR you are saying: "Here is my branch. Please compare it to main, show the diff, let people comment, and then merge it when it's approved."

The review step is the point. PRs are how teams catch bugs, share knowledge, and keep the main branch stable. Even on solo projects they are useful — the act of writing a PR description often surfaces problems you missed.

Merge Conflicts — What They Are and Why They Happen

Git can usually merge automatically

When you merge two branches, Git compares the changes line by line. If two people edited different lines, Git figures it out silently and everything just works. Most merges are automatic.

A merge conflict happens when two branches both changed the same line in different ways. Git has no way to know which version is correct — so it stops and asks you to decide.

mainmotto A
conflict-setupmotto B
both changed the same line
→ merge →
CONFLICTyou decide

When two branches change the same line, Git produces a conflict

What the conflict markers look like

Git inserts markers directly into the file showing both versions:

<<<<<<< HEAD (conflict-setup)
motto = "if it compiles, ship it"
=======
motto = "we build things that work"
>>>>>>> main
  • Everything between <<<<<<< and ======= is your branch's version
  • Everything between ======= and >>>>>>> is the incoming version

To resolve it: delete the markers, keep whichever version (or both, or something new), save the file, then stage and commit.

VS Code makes this visual. When you open a conflicted file, VS Code highlights the two versions with colored backgrounds and shows buttons — "Accept Current Change", "Accept Incoming Change", "Accept Both". Click one and VS Code removes the markers for you. Then just save and commit.

Part 1 — Open a Pull Request

Add your name to a shared file using the branch → commit → PR workflow

Everyone in the class (including the instructor) will add one line to exercises/pr-and-conflicts/shared-file.py. By the end, the PR list on GitHub will show a PR per person — all changing the same file, all merged cleanly.

  • Clone the repo (if you haven't already) 
    git clone https://github.com/jbain-ANDREW/Training-Repo.git
cd Training-Repo
  • Create your branch

    Never work directly on main. Replace with your actual name in lowercase:

    git checkout -b firstname-lastname
  • Edit shared-file.py

    Open exercises/pr-and-conflicts/shared-file.py in VS Code. Find the participants list and add one line at the end — your name and one word that describes you:

    participants = [
    "James Bain  : builder",        # instructor
    "Firstname Lastname : your_word",  # ← add this line
]

    Do not edit anyone else's line. Keep the same format.

  • Stage and commit 
    git add exercises/pr-and-conflicts/shared-file.py
git commit -m "Add Firstname Lastname to participants"
  • Push your branch 
    git push origin firstname-lastname
  • Open a Pull Request on GitHub

    Go to github.com/jbain-ANDREW/Training-Repo. GitHub will show a yellow banner: "firstname-lastname had recent pushes — Compare & pull request." Click it. Set base to main, compare to your branch, write a short title, and submit.

    Your PR will appear in the open PRs list. Wait for your instructor to review and merge it.
If your PR shows a conflict: Someone else was merged first and their entry is now in the file too. Update your branch: 
git checkout firstname-lastname
git pull origin main --rebase
git push origin firstname-lastname --force-with-lease
This replays your commit on top of the latest main — the conflict disappears.

Part 2 — Resolve a Merge Conflict

Check out a branch that deliberately diverges from main — then merge and fix the conflict

The conflict-setup branch was created with a different motto line than main. When you merge main into it, Git cannot decide which motto is right — so it produces a conflict and asks you to choose.

  • Fetch the remote branch and check it out 
    git fetch origin
git checkout conflict-setup

    You are now on the conflict-setup branch, which has motto = "if it compiles, ship it".

  • Merge main into it 
    git merge main

    Git will stop with a message like:

    Auto-merging exercises/pr-and-conflicts/shared-file.py
CONFLICT (content): Merge conflict in exercises/pr-and-conflicts/shared-file.py
Automatic merge failed; fix conflicts and then commit the result.
  • Open the conflicted file in VS Code

    VS Code will highlight the conflict zone. You will see both motto versions separated by markers. Choose one — or write something entirely new — then delete all three marker lines (<<<<<<<, =======, >>>>>>>) and save the file.

    VS Code shortcut: Click "Accept Current Change" to keep the conflict-setup version, or "Accept Incoming Change" to take the main version. Either is fine — the point is to practice resolving.
  • Stage the resolved file and finish the merge 
    git add exercises/pr-and-conflicts/shared-file.py
git commit -m "Resolve motto conflict"

    Git will use a default merge-commit message — that is fine. Save and close the editor if it opens one.

  • Push and open a Pull Request 
    git push origin conflict-setup

    Go to GitHub and open a PR — base: main, compare: conflict-setup. In the description, note which motto you kept and why.

You just did what every developer does dozens of times a week. Merge conflicts look scary the first time. After three or four of them, they become a normal five-minute task.

Quick Reference

PR workflow in four commands

git checkout -b my-branch       # create a branch
git add <file>                  # stage changes
git commit -m "description"     # commit
git push origin my-branch       # push — then open PR on GitHub

Resolving a conflict in four steps

git merge <branch>              # triggers the conflict
# edit the file — remove <<< === >>> markers, keep the right code
git add <file>                  # mark as resolved
git commit                      # finish the merge

Check what git thinks is happening

git status                      # shows conflicted files with "both modified"
git diff                        # shows the full diff including conflict markers
Abort a merge you're not ready to handle: git merge --abort resets everything back to before the merge started.
