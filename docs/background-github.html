The Problem GitHub Solves

Imagine you're working on a document — a report, a website, a script. At some point you make a change that breaks something. You want to go back to how it was yesterday, but you've been overwriting the same file. Or you're collaborating with someone and you both make different changes to the same file at the same time. Who wins?

Before version control tools existed, people handled this with naming conventions — report_v2.docx, report_v2_final.docx, report_v2_final_REAL.docx. It works until it doesn't.

✗ Without version control

  • Files named final_v3_ACTUAL.html
  • Can't tell what changed or when
  • Emailing zip files back and forth
  • Accidentally overwriting someone's work
  • No way to safely experiment

✓ With version control (Git + GitHub)

  • One file, full history of every change
  • Every change tagged with what, who, and when
  • Roll back to any previous state instantly
  • Multiple people work in parallel safely
  • Experiment freely — branches are cheap

Git vs. GitHub — What's the Difference?

Git is the version control software itself — it runs on your computer and keeps track of changes to your files. It was created by Linus Torvalds (the same person who created Linux) in 2005. Git works entirely locally, even with no internet connection.

GitHub is a website (and service) that stores your Git repositories in the cloud. Think of Git as the engine and GitHub as the garage where the engine stores its work so others can see it, contribute, and collaborate.

Your computerGit tracks changes locally
→ git push →
GitHubCloud backup + sharing
→ git clone →
CollaboratorGets a full copy

Git handles the tracking; GitHub handles the sharing.

GitHub is not the only option — GitLab and Bitbucket do the same thing. But GitHub has the largest community and the most open-source projects, so it has become the default.

Key Concepts in 60 Seconds

Repository (repo)

A folder that Git is tracking. It contains your project files plus a hidden folder (.git/) where Git stores the entire history of every change ever made. You never edit the .git/ folder directly — Git manages it.

Commit

A saved snapshot of your project at a specific moment. Every commit has a message describing what changed, plus a timestamp and the author's name. You can always return to any commit.

→
Commit BAdded header
→
Commit CFixed typo
→
Commit DYou are here

Every commit is a checkpoint you can return to at any time.

Branch

A parallel version of the project where you can experiment without affecting the main version. When your experiment works, you merge it back. This is how teams work simultaneously on different features without stepping on each other.

Pull Request (PR)

A request to merge your branch into another branch (usually main). It's also a review mechanism — teammates can look at the changes, leave comments, and approve before anything gets merged. In this course, you'll open a PR to submit your exercise.

Fork

A personal copy of someone else's repository on your own GitHub account. You can make any changes you want to your fork without affecting the original. It's how you contribute to projects you don't have direct write access to.

Why GitHub Became the Standard

GitHub launched in 2008 and grew by hosting open-source projects for free. By the time major companies needed a collaboration platform, GitHub already had millions of projects, packages, and developers. The network effect kicked in: more projects meant more developers, which attracted more projects.

Today GitHub hosts over 400 million repositories. Most open-source libraries you'll ever install live there. Job listings for software roles routinely ask for a GitHub profile link the way design roles ask for a portfolio. Microsoft acquired GitHub in 2018 for $7.5 billion — and kept it free for most users.

Learning Git and GitHub is not optional if you intend to work with code professionally. It's the infrastructure that everything else plugs into.

Getting a GitHub Account

A basic GitHub account is free and takes two minutes to create. Go to github.com, click Sign up, and follow the steps. Choose a username you'd be comfortable showing to a future employer — your GitHub profile is often your first technical portfolio.

Students: After creating your account, apply for the GitHub Student Developer Pack at education.github.com. It gives you free access to GitHub Copilot and dozens of other tools normally behind paywalls.

Public vs. Private Repositories

Public repos are visible to anyone on the internet — no account required to read or download them. The Training-Repo you'll be working with is public. Anyone can clone it without logging in.

Private repos are only visible to you and people you explicitly invite. If you want to keep your work private, you can create a private fork — but for this course, public is fine and makes submission easier.

You don't need to share your GitHub username with the instructor to access the repo — it's public. But please do share it anyway so the instructor can find your submitted work easily. See the submission instructions below.

How to Submit Your Exercise Work

When you complete an exercise, here's the full submission workflow:

  1. Fork the Training-Repo

    On the repo page, click the Fork button in the top-right corner. This creates your own copy of the repo under your GitHub account.

  2. Clone your fork to your computer

    Use the GitHub exercise instructions to clone your fork locally. You'll make changes here.

  3. Complete the exercise and commit your changes

    Follow the exercise steps, then commit and push your changes to your fork on GitHub.

  4. Open a Pull Request

    Go to your fork on GitHub and click Contribute → Open Pull Request. This signals to the instructor that your work is ready to review.

  5. Email your instructor

    Send a short email to let the instructor know you've submitted, including your GitHub username and a link to your pull request. This ensures nothing slips through the cracks.

How the Instructor Reviews Your Work

GitHub shows exactly what changed between your version and the original — line by line. The instructor can see every file you added or modified, leave comments on specific lines, and follow the full history of what you did and when. You don't need to explain what you changed; the commit history does that automatically.
